# Liquid Cooling Leak Detection in SONiC BMC

## 1. Overview

Due to the excessive heat generated by the equipment, traditional air-cooling methods are no longer sufficient for effective heat dissipation. Therefore, liquid cooling technology has become a necessary choice for more efficiently cooling the equipment and ensuring its proper operation. Given the potential fatality of liquid cooling leakage, implementing a mechanism to monitor and instantly alert the system when such an event occurs is crucial. This document describes the SONiC BMC platform APIs, PMON responsibilities, database schema, and user-visible status for system leak detection on platforms where `chassis.is_liquid_cooled()` is true.

## 2. Requirements

1. Monitor leak sensors only on liquid-cooled or hybrid-cooled platforms.
2. Detect per-sensor leak state and per-sensor leak severity.
3. Aggregate per-sensor results into a system leak severity.
4. Expose per-sensor and system leak status through Redis DB for BMC policy enforcement and CLI.
5. Avoid additional PMON overhead on platforms that are not liquid cooled.

## 3. Detection and Action Flow

This section is a high-level summary of the leak handling flow described in [pmon-bmc-design.md](sonicBMC/pmon-bmc-design.md). The PMON/BMC design document is the source of truth for the detailed handling process and daemon interactions.

System leak handling is split between `thermalctld` and `bmcctld`.

1. `thermalctld` runs a dedicated leak-monitoring thread and polls the platform leak sensor APIs.
2. The same `LiquidCoolingUpdater` thread evaluates per-sensor health, leak state, and per-sensor severity.
3. The thread updates per-sensor state in `LIQUID_COOLING_INFO|leakage_sensors{X}`.
4. The thread aggregates the current sensor results into the system leak severity.
5. The same thread writes the aggregate result to `SYSTEM_LEAK_STATUS|system`.
6. `bmcctld` subscribes to `SYSTEM_LEAK_STATUS|system` and applies the configured action from `LEAK_CONTROL_POLICY`.
7. CLI commands and other consumers read the database state to show current leak status.

Redis DB is used here as an inter-process state and notification mechanism for consumers such as `bmcctld`, CLI, and legacy status readers.

The BMC remains powered on during leak events. Actions taken on the Switch-Host depend on the configured policy.

## 4. Platform APIs

The leak-detection API is implemented in `sonic-platform-common` and exposed through the platform objects below.

### 4.1 LeakageSensorBase

`LeakageSensorBase` provides the per-sensor leak interface used by `thermalctld`.

| Method | Description |
|--------|-------------|
| `get_name()` | Returns the leak sensor name |
| `is_leak()` | Returns whether a stable leak condition is detected |
| `is_leak_sensor_ok()` | Returns whether the sensor is healthy or faulty |
| `get_leak_sensor_type()` | Returns the sensor type, for example `rope`, `spot`, or `flex_pcb` |
| `get_leak_sensor_location()` | Returns the physical sensor location |
| `get_leak_severity()` | Returns the per-sensor leak severity as `LeakSeverity.CRITICAL`, `LeakSeverity.MINOR`, or `None` |
| `get_leak_profile()` | Returns the `LeakSensorProfileBase` associated with this sensor type |

`get_leak_severity()` is platform specific. A sensor can be marked `CRITICAL` because of location, for example near ASIC or CPU components, or because the leak extent indicates a severe event.

### 4.2 LeakSensorProfileBase

`LeakSensorProfileBase` provides per-sensor-type policy values used during leak aggregation.

| Method | Description |
|--------|-------------|
| `get_type()` | Returns the leak sensor type described by this profile |
| `get_leak_max_minor_duration_sec()` | Returns the maximum time a `MINOR` leak may persist before it is treated as `CRITICAL` |

### 4.3 LiquidCoolingBase

`LiquidCoolingBase` exposes the liquid-cooling device view used to enumerate sensors and profiles.

| Method | Description |
|--------|-------------|
| `get_num_leak_sensors()` | Returns the number of leak sensors |
| `get_leak_sensor(index)` | Returns a specific leak sensor object |
| `get_leak_sensor_status()` | Returns the list of sensor objects currently reporting a leak |
| `get_all_leak_sensors()` | Returns the list of all leak sensors |
| `get_all_profiles()` | Returns the list of all leak sensor profiles |
| `get_profile(type)` | Returns the profile associated with the given sensor type |

### 4.4 ChassisBase

`ChassisBase` exposes the platform capability check needed to enable leak monitoring only on supported systems.

| Method | Description |
|--------|-------------|
| `is_liquid_cooled()` | Returns whether the chassis is liquid cooled or hybrid cooled |

## 5. thermalctld Design

On platforms where `chassis.is_liquid_cooled()` is false, leak monitoring is skipped entirely.

On liquid-cooled platforms, `thermalctld` has a dedicated `LiquidCoolingUpdater` thread for leak detection. That thread:

1. Polls the platform leak sensors.
2. Updates per-sensor state in `LIQUID_COOLING_INFO`.
3. Computes the aggregate system leak severity.
4. Updates `SYSTEM_LEAK_STATUS|system` when the aggregate state changes.

The dedicated thread exists because leak events are rare but time sensitive, and they should not be limited by the slower main thermal polling cadence.

An implementation may use a short configurable polling interval for the dedicated leak-monitoring thread.

### 5.1 Per-Sensor Update Flow

This pseudocode is illustrative and summarizes the `thermalctld` leak-update flow at a high level. The detailed daemon workflow remains defined in [pmon-bmc-design.md](sonicBMC/pmon-bmc-design.md).

```python
class LiquidCoolingUpdater:
    def update(self):
        self._refresh_leak_status_update()

    def _refresh_leak_status_update(self):
        if not chassis.is_liquid_cooled():
            return

        liquid_cooling = <platform LiquidCoolingBase instance>
        for sensor in liquid_cooling.get_all_leak_sensors():
            update_liquid_cooling_info(sensor)

        system_severity = calculate_system_leak_severity()
        update_system_leak_status(system_severity)
```

### 5.2 System Leak Severity Algorithm

This section is a high-level summary of the system leak severity algorithm described in [pmon-bmc-design.md](sonicBMC/pmon-bmc-design.md). The PMON/BMC design document remains the source of truth for the detailed policy definition.

The `LiquidCoolingUpdater` thread applies the following aggregation logic using its in-memory view of the current sensor states.

| Individual Leak Sensor Condition | Individual Leak Sensor Severity | System Leak Severity |
|----------------------------------|---------------------------------|----------------------|
| 1 critical leak | `CRITICAL` | `CRITICAL` |
| 2 or more leaks | Any severity | `CRITICAL` |
| 1 minor leak staying for MAX-T seconds | `MINOR` | `CRITICAL` |
| 1 minor leak detected | `MINOR` | `MINOR` |

`MAX-T` is provided by the leak sensor profile through `max_minor_duration_sec`.

## 6. Redis DB Schema

### 6.1 Per-Sensor Leak State

`thermalctld` stores per-sensor leak state in `STATE_DB`.

```text
key                       = LIQUID_COOLING_INFO|leakage_sensors{X}
; field                   = value
name                      = STR
leaking                   = STR        ; Yes, No, or N/A if the sensor is faulty or unreadable
leak_status               = STR        ; Same value as leaking, kept for backward compatibility
leak_sensor_status        = STR        ; Good or Fault
type                      = STR        ; sensor type from get_leak_sensor_type()
location                  = STR        ; sensor location from get_leak_sensor_location()
leak_severity             = STR        ; CRITICAL, MINOR, or None
```

### 6.2 Leak Profiles

`thermalctld` publishes leak sensor profiles to `STATE_DB` at startup.

```text
key                       = LEAK_PROFILE|<sensor_type>
; field                   = value
type                      = STR
max_minor_duration_sec    = integer
```

### 6.3 Aggregate System Leak State

The system leak result is stored in `STATE_DB` and consumed by `bmcctld`.

```text
key                       = SYSTEM_LEAK_STATUS|system
; field                   = value
device_leak_status        = STR        ; CRITICAL, MINOR, or None
timestamp                 = STR
```

## 7. bmcctld Policy Enforcement

This section summarizes how `bmcctld` consumes system leak state and applies policy. The detailed event handling and state-transition behavior are defined in [pmon-bmc-design.md](sonicBMC/pmon-bmc-design.md).

`bmcctld` subscribes to `SYSTEM_LEAK_STATUS|system` and applies the configured action from `LEAK_CONTROL_POLICY`.

- `system_critical_leak_action` defaults to `power_off`
- `system_minor_leak_action` defaults to `syslog_only`

Supported actions are:

- `syslog_only`
- `graceful_shutdown`
- `power_off`

If the corresponding leak policy is disabled, `bmcctld` logs the event and skips enforcement.

## 8. CLI Management

This section provides a high-level summary of the leak-related CLI surface. The PMON/BMC design document remains the source of truth for command definitions and detailed output behavior.

### 8.1 `show platform leak status`

This command displays per-sensor leak state from `LIQUID_COOLING_INFO`.

```text
show platform leak status
Name             Leak  Leak-sensor-status leak-sensor-type leak-severity
---------      ------- ------------------- ---------------- -------------
leak_sensors1    YES         OK                 rope           MINOR
leak_sensors2    NO          FAULTY             spot           NA
...
leak_sensorsX    YES         OK                 flex_pcb       CRITICAL
```

### 8.2 `show platform leak profiles`

This command displays the configured leak profiles.

```text
show platform leak profiles
Sensor-Type    Max-Minor-Duration-Sec
-----------    ----------------------
rope                        300
spot                        600
flex_pcb                    180
```

### 8.3 `show platform leak control-policy`

This command displays the active leak-control policy used by `bmcctld`.

```text
show platform leak control-policy
system_leak_policy              : enabled
system_critical_leak_action     : power_off
system_minor_leak_action        : syslog_only
rack_mgr_leak_policy            : enabled
rack_mgr_critical_alert_action  : syslog_only
rack_mgr_minor_alert_action     : syslog_only
```

### 8.4 `show system-health detail`

System health monitoring also consumes `LIQUID_COOLING_INFO` and evaluates leak sensors as part of the hardware health view. The hardware checker uses the `leak_status` field to determine whether each leak sensor should be reported as `OK` or `Not OK`. When a sensor starts reporting a leak or recovers from a leak condition, system health also publishes the corresponding liquid-cooling event.

```text
show system-health detail
System services and devices monitor list
Name                     Status    Type
-----------------------  --------  -------------
...
leak_sensors1            OK        LiquidCooling
leak_sensors2            OK        LiquidCooling
...
leak_sensors3            Not OK    LiquidCooling
```

## 9. Performance

Leak monitoring runs only on platforms where `chassis.is_liquid_cooled()` is true. A dedicated thread is used so leak detection can react quickly without forcing the main thermal polling loop to run at the same cadence.

## 10. Testing

Mock testing should cover both per-sensor updates and aggregate system behavior. At minimum:

1. Verify per-sensor data is written correctly to `LIQUID_COOLING_INFO`.
2. Verify the system leak severity is written correctly to `SYSTEM_LEAK_STATUS|system`.
3. Verify `bmcctld` takes the configured action for `MINOR` and `CRITICAL` system leak status.
4. Verify the relevant syslog messages are emitted.
5. Verify `show platform leak status` output is correct.
6. Verify `show platform leak profiles` output is correct.
7. Verify `show system-health detail` output includes the expected leak sensor entries.
8. Verify system health publishes leak and recovery events for leak sensor state transitions.
